# Jira CSV Generation Guidelines

## Purpose

These guidelines define how to generate Jira Tasks and subtasks that operationalize active marketing campaign Epics for a given month. The output is a Jira-importable CSV that creates Tasks, descriptions, subtasks, and dates under existing Epics.

Jira becomes the operational record for what shipped and when ŌĆö making "what went out last week" answerable directly from the board.

## Inputs Required

Each run requires:

1. Target timeframe
2. Timeframe-specific context (product launches, events, announcements)

## Jira Structure

### Epics

Epics represent a campaign theme and are long-lived. They are assumed to exist before task generation and are not included in the CSV.

Naming format: `YEAR Theme`
Example: `2026 Foundations Free`

**Current Epic key (for scaffold pre-fill):** `MCM-1139` ŌĆö update this line when the campaign Epic changes; the scaffold reads it rather than guessing.

### Tasks

Tasks are the atomic unit of work. Each Task represents one publishable deliverable ŌĆö completion equals publication.

Naming format: `Lever - Channel - Deliverable Name`
Example: `Social Media - LinkedIn - Foundations Announcement`

Tasks include an auto-generated description with sample text based on the campaign theme and context.

### Subtasks

Subtasks represent production steps and dependencies within a Task.

Naming format: `Channel - Step`
Example: `LinkedIn - Image Creation`

Subtasks include an auto-generated action description.

### Short video ŌĆö out-of-band

As of 2026-06-01, LinkedIn short videos are planned and produced outside the content-plan system. The weekly Jira CSV still includes a short-video Task for capacity tracking, but it is **barebones**:

- **Summary:** `Social Media - LinkedIn - Short Video (out-of-band)`
- **Description:** one line ŌĆö `Short video ŌĆö topic, wrap, and copy planned outside the content-plan system. This row exists for Jira capacity tracking only. Furqan fills topic + asset at publish time.`
- **Subtasks:** none. The Task is the unit of work.
- **No** `Slug:`, `Wraps:`, `Copy:`, or `Post type:` lines ŌĆö these belong to the wrap pattern, and the short video no longer wraps a planned blog by default.
- Start/Due dates still come from the slate row.

The weekly slate marks the short-video row as `ŌĆö (planned out-of-band)`.


## Date Rules

**Start date** is the date work begins on the first production subtask.

**Due date** is the publish date. All subtasks must complete on or before the Task due date.

Social media posts publish Tuesday through Friday by default. All other channels publish on the earliest eligible weekday unless otherwise constrained. Calendar constraints override all default publish rules. Each Task has exactly one publish date.

## Channel Subtask Templates

Subtasks are attached automatically based on channel. No additional subtasks are generated by default.

Description must include: platform (cerkl.com vs. internalcommspro.com), topic, prompt to generate a draft, theme context.

**LinkedIn specifics (applied by `jira/jira-scaffold-process.md`):**

- The LinkedIn **Task** Description ends with a `Copy:` line followed by a `[COPY_PLACEHOLDER]` token ŌĆö the LinkedIn drafting process replaces it (on the parent Task, not in a subtask) with the drafted caption + asset spec + hashtags. The `LinkedIn ŌĆō Copy` subtask keeps its plain action description (`Draft the post copy.`); it is the production step, not where the copy is stored.
- For `poll` post-type rows, the `LinkedIn ŌĆō Asset Creation` subtask Description reads: `(N/A for polls ŌĆö native poll widget; skip or mark done.)`. The subtask still exists for shape consistency; it just carries the explicit skip note.
- **Short video exception:** short-video Tasks are barebones ŌĆö no subtasks, no `Copy:` block, no `Wraps:` line. See [Short video ŌĆö out-of-band](#short-video--out-of-band) above. The 4 standard subtasks (Copy / Asset Creation / Approval / Implementation) are NOT generated for short-video rows.

## Jira Field Mapping

### Column order (canonical)

Every CSV uses this column order, left to right:

```
Work Item ID, Issue Type, Parent ID, Epic Link, Summary, Description, Owner, Channel, Start Date, Due Date
```

`Work Item ID` and `Parent ID` are **CSV-local placeholder IDs** Jira uses at import time to stitch Tasks to their Subtasks. They never appear in Jira after import ŌĆö Jira assigns real issue keys. Convention:

- `T001`, `T002`, `T003`, ŌĆ” ŌĆö Task rows. Renumber per CSV (each weekly CSV starts at `T001`).
- `S001`, `S002`, `S003`, ŌĆ” ŌĆö Subtask rows. Renumber per CSV (each weekly CSV starts at `S001`).
- A Subtask's `Parent ID` cell holds the parent Task's `Work Item ID` (e.g., `T001`).
- Tasks leave `Parent ID` blank.

### Task Fields

| Field | Value |
|---|---|
| Work Item ID | `T001`, `T002`, ŌĆ” |
| Issue Type | Task |
| Parent ID | (blank) |
| Epic Link | Pre-filled at scaffold time with the current campaign Epic key (override at import if it varies) |
| Summary | `Lever - Channel - Deliverable Name` |
| Description | Generated block including `Slug: <slug>`, `Draft (Google Doc): [DRIVE_URL_PLACEHOLDER]`, and any brief/keyword/CMS metadata |
| Owner | Channel owner Jira ID (see Ownership table) |
| Channel | Custom Channel field |
| Start Date | Date work begins |
| Due Date | Publish date |

### Subtask Fields

| Field | Value |
|---|---|
| Work Item ID | `S001`, `S002`, ŌĆ” |
| Issue Type | Subtask |
| Parent ID | The parent Task's `Work Item ID` (e.g., `T001`) |
| Epic Link | (blank ŌĆö inherited from parent at import) |
| Summary | `Channel - Step` |
| Description | Generated description based on theme and task topic |
| Owner | Pre-filled at scaffold time with the channel owner's Jira ID (override at import when a subtask owner varies) |
| Channel | Same as parent's Channel |
| Start Date | When this subtask begins (typically before the parent Task's Due Date) |
| Due Date | When this subtask completes (Ōēż parent Task Due Date) |

## CSV Output Requirements

- One row per Task and subtask
- `Work Item ID` and `Parent ID` placeholders link rows; Jira resolves them at import
- Dates in Jira-compatible format (YYYY-MM-DD)
- Multi-line `Description` fields use standard CSV quoting (double-quoted, embedded newlines preserved)
- Importable without manual cleanup ŌĆö `Epic Link` and subtask `Owner` are pre-filled at scaffold time; override at import only when something varies

## Reusability

These guidelines apply to every Epic and campaign without modification. Any deviation is an explicit, ad hoc override ŌĆö not an update to the system.
