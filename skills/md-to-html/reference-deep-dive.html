The Unreasonable Effectiveness of HTML

Date: 2026-05-10 Category: Anthropic prompting / output-format technique Worth trying — Yes

Anthropic engineer Thariq Shihipar argues you should ask Claude to output HTML (not Markdown) for any substantive artifact: plans, code reviews, design docs, dashboards. The cost is one extra sentence in the prompt (and ~1.2× more tokens); the payoff is interactive, denser, link-shareable artifacts that Markdown can't represent. Production users report real wins replacing custom render frameworks with raw HTML output.

Try this in your next deep-dive run
Append this exact line to the synthesis brief in cc-tool-deepdive-process.md:
Output HTML, neatly styled and using capabilities of HTML and CSS and JavaScript to make the explanation rich and interactive and as clear as possible.

01What it is & what it does

Origin

A prompting technique published by Thariq Shihipar (Claude Code team, Anthropic) on 2026-05-08 and amplified by Simon Willison's weblog the same day. Comes with a companion site of 20 self-contained HTML artifacts produced by Claude Code, each illustrating a real use case.

  • Original (Thariq): thariqs.github.io/html-effectiveness
  • Simon Willison's writeup: simonwillison.net
  • Community skill: github.com/dogum/html-artifacts
  • HN thread (500+ pts, 269 comments): news.ycombinator.com/item?id=48071940

How you actually use it

You append one line to your prompt asking for HTML. Claude returns a single self-contained HTML file with inline styles + JS. You open it in a browser (or a plain-HTML preview pane in the IDE) instead of reading rendered Markdown.

The companion site shows what these artifacts look like — they aren't styled blog posts; they're interactive documents with sortable tables, sliders, copy buttons, and SVG diagrams.

Thariq's five pillars

PILLAR 1

Density

Tables, SVG, CSS, JS — all in one file.

PILLAR 2

Readability

Past ~100 lines, Markdown becomes unreadable. HTML scales.

PILLAR 3

Sharing

A link beats a Markdown attachment.

PILLAR 4

Interactivity

Sliders, knobs, copy-to-prompt buttons, collapsibles.

PILLAR 5

Joy factor

Outputs feel like artifacts, not text dumps.

02Who it's for

Primary user

Anyone using Claude or Claude Code to produce documents that will be read by humans, not parsed by code.

Strong fit when

Plan docs, code reviews, architecture proposals, comparison matrices, status reports, custom mini-editors, technical explainers, dashboards over local data (DuckDB, CSVs).

Weak fit when

Short conversational replies, terminal-only context (no rendering surface), code-only outputs, anything you'll re-process programmatically.

03How it slots into Travis's stack

Three immediate fits:

cc-trends daily recap

The recap is a multi-section document with categorized lists, dates, links, "Why this matters" callouts. HTML output would let it carry color-coded categories, collapsible sections, and an inline filter bar. Tradeoff: render to .html on disk, lose the IDE's Markdown preview.

Tool deep-dive output

Exactly the artifact type Thariq enumerates. Replace this very file's Markdown with HTML and you get: live comparison-matrix, collapsible Gotchas, click-to-copy install commands. Highest-leverage place to try it.

Cerkl marketing reports

If Travis ever generates briefings for stakeholders, HTML artifacts beat both Markdown and Slides because they're shareable as a single link with no Slides license.

What you'd have to change: pick a render-and-host pattern. Options: (a) save to local .html files and open in Chrome (zero infra, ugly URLs), (b) render to ~/Documents/Claude/artifacts/ served by a tiny local HTTP daemon, (c) push to a private gist or GitHub Pages branch for share links.

04Comparison to alternatives

AlternativeHow they differ
Markdown (default) Caps out at headers/lists/tables/bold; can't express layout, color coding, interactivity, collapsibles. Best for short conversational replies.
XML tags Anthropic's structuring primitive — but for input delimiting, not rendered output. No native browser styling. docs
JSON / structured outputs Maximally machine-parseable; expresses data, not documents. Useless for human-facing rich display.
Tool-use / function-calling Schema-constrained outputs for downstream code. Different problem entirely — reliability for programs, not richness for humans. docs
Decision-relevant axis: render fidelity for humans. HTML is the only one that goes to a browser and arrives interactive.

05Gotchas

Token cost
~1.2× vs. Markdown for the same content (HN: abirch). A WYSIWYG-flavored HTML output can be much higher with <span> soup.
Render surface required
Useless in plain terminal contexts. Travis's IDE renders Markdown but won't render HTML inline; you need to open the file in Chrome or use a preview pane (e.g., VS Code Live Preview).
Script-injection risk (overlooked in the discussion)
HTML output can include <script> tags. Thariq's prompt doesn't sandbox the JS. If you're going to publish these or render where a third party sees them, treat the JS as untrusted: strip <script> or render in a sandboxed iframe.
Note on the dichotomy
Markdown already permits inline HTML/SVG (HN: PhilippGille, mock-possum) — so you can ask Claude to emit Markdown with inline <svg> and <details> for a hybrid.

07Recommendation

Worth trying? Yes — for deep-dive outputs and the cc-trends daily recap specifically.

First thing to try

Re-emit this very deep-dive as HTML on the next run. Append the magic sentence to the synthesis brief in cc-tool-deepdive-process.md. Reversible in one line.

Time-box

One deep-dive run + one cc-trends daily run, both compared side-by-side against their Markdown counterparts. ~30 min of attention.

What would change my mind

If rendering ergonomics suck (no good preview, opening in Chrome breaks flow), or if the token-cost increase shows up as recap latency. If the HTML versions feel like artifacts you actually revisit while the Markdown ones rot — that's the win.

08Open questions

Best preview surface for VS Code?
ms-vscode.live-server (Microsoft Live Preview) is now installed. Worth comparing against just opening the file in Chrome side-by-side with the IDE. ~10 min to decide.
Skill or per-prompt?
Cleaner to wrap the magic sentence in a small Claude Code skill (e.g., output-html) that any process can invoke. The community already has dogum/html-artifacts — worth reading their SKILL.md before rolling your own.
Stable URL pattern for share links?
If artifacts go to a private gist, that's friction; if local-only, no link. The sharing pillar only pays off with a render-and-host pipeline, which doesn't exist in your stack yet.
Does it compose with Outcomes / multiagent sessions?
Code w/ Claude shipped Outcomes — does an Outcome rubric work better when the agent produces HTML vs. Markdown? Untested.
